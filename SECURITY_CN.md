## 报告漏洞

如果你在 Rausu 中发现安全漏洞，请负责任地报告。

**请勿公开提交 GitHub Issue 来报告安全漏洞。**

### 如何报告

1. 发送邮件至：**security@rausu.dev**（或创建 [私有安全公告](https://github.com/ypwu1/rausu/security/advisories/new)）
2. 请包含：
   - 漏洞描述
   - 复现步骤
   - 受影响版本
   - 潜在影响评估
   - 建议的修复方案（如有）

### 预期流程

- **确认收到**：48 小时内
- **初步评估**：7 天内
- **修复计划**：14 天内沟通
- **致谢**：在安全公告中致谢（除非你希望匿名）

### 范围

以下属于安全报告的范围：

- 认证/授权绕过（虚拟 Key 系统）
- 远程代码执行
- 路径穿越 / 目录遍历
- 通过 provider URL 的服务端请求伪造（SSRF）
- API Key 泄露（日志、响应、错误信息）
- 信息泄露（内部状态、配置密钥）
- 资源耗尽导致的拒绝服务
- 代理请求走私 / Header 注入
- 传输或存储中的凭证暴露
- Provider 凭证越权（访问 Key 作用域之外的 provider）

## 安全架构

Rausu 在代理管道的每一层实施纵深防御。

### API Key 安全
- **哈希存储**：虚拟 Key 以 SHA-256 哈希形式存储——从不明文
- **密钥脱敏**：API Key 永远不会出现在日志中，即使在 `trace` 级别
- **内存安全**：Rust 的所有权模型防止 use-after-free 和缓冲区溢出
- **零 `unsafe` 块**：核心代理代码不使用 `unsafe` Rust

### 输入验证
- **请求验证**：所有入站请求在转发前都经过 OpenAI schema 验证
- **Header 净化**：上游 provider 凭证在服务端注入；客户端提供的认证 header 不会被转发
- **大小限制**：请求体大小限制防止内存耗尽攻击
- **JSON 解析**：严格反序列化——畸形 payload 被拒绝，不会被隐式转换

### 网络安全
- **TLS 支持**：可选的 TLS 终止，支持无反向代理的直接部署
- **CORS 策略**：可配置的 CORS，默认限制性策略
- **安全 Header**：每个响应都包含 `X-Content-Type-Options`、`X-Frame-Options`、`X-Request-Id`
- **Provider 隔离**：Provider 凭证有作用域限制——被泄露的虚拟 Key 无法访问其他 provider 的 API Key
- **超时强制**：每个 provider 的请求超时防止挂起连接耗尽资源

### 审计与可观测性
- **请求日志**：每个代理请求都记录请求 ID、模型、provider 和状态（请求体可选）
- **错误上下文**：Provider 错误记录完整上下文，但敏感字段已脱敏
- **结构化日志**：JSON 格式日志，可集成 SIEM 和日志聚合系统

### 速率限制（Phase 3）
- **按 Key 限制**：每分钟请求数（RPM）和每分钟 token 数（TPM）
- **预算强制**：按 Key、团队、用户的硬限制和软限制
- **熔断器**：重复失败时自动禁用 provider

### 安全防护（Phase 4）
- **内容过滤**：基于模式匹配的请求/响应拦截
- **PII 脱敏**：在转发到 provider 之前检测并删除个人信息
- **提示注入检测**：对常见注入模式的启发式扫描
- **中间件管道**：有序的防护链，包含请求前和响应后钩子

## 依赖项

安全关键依赖：

| 依赖 | 用途 |
|------|------|
| `axum` | 内置提取验证的 HTTP 服务器 |
| `reqwest` | 支持 TLS 的 HTTP 客户端（rustls/native-tls） |
| `serde` | 安全的序列化/反序列化 |
| `tracing` | 无格式字符串漏洞的结构化日志 |
| `tokio` | 有界任务生成的异步运行时 |
| `uuid` | 请求 ID 生成（v4 随机） |

### 依赖策略

- 所有依赖通过 `Cargo.lock` 锁定
- CI 中运行 `cargo audit` 检测已知漏洞
- 最小依赖树——更少的依赖 = 更小的攻击面
- 核心代理无 C 依赖（尽可能纯 Rust）

## 加固建议

生产部署建议：

1. **运行在反向代理后面**（nginx、Caddy）用于 TLS 终止和额外的速率限制
2. **使用环境变量**存放所有密钥——永远不要将 API Key 提交到配置文件
3. **设置强 master key**——使用 `openssl rand -hex 32` 或等效方法
4. **启用请求日志**到外部日志聚合系统
5. **限制网络访问**——Rausu 应只对授权客户端可达
6. **监控费用**——配置预算告警以尽早发现被泄露的 Key
7. **定期轮换 Key**——虚拟 Key 和上游 provider 凭证都要轮换
8. **锁定版本**——使用具体的版本 tag，而不是 `latest`
